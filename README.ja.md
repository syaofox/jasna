# Jasna

### VRAM 管理
Jasna は VRAM を自動的に管理します。GPU の VRAM が不足すると、処理キュー内のフレームが一時的にシステム RAM に移動され、必要な時に戻されます。これはバックグラウンドで行われ、設定は不要です。

それでもメモリ不足エラーが発生する場合は、**クリップサイズ**を縮小（例：180 から 60）するか、モデルコンパイルを無効にしてください。どちらもピーク VRAM を大幅に削減します（下表参照）。

### 使い方
最新のリリースパッケージ（Windows/Linux）をダウンロードしてください。

- **アプリをダウンロードした場合（推奨）**:
  - **Windows**: そのまま使えます — Jasna には必要なもの（`ffmpeg`、`ffprobe`、`mkvmerge`）がすべて含まれています。
  - **Linux**: `ffmpeg`、`ffprobe`（**メジャーバージョンは 8 である必要があります**）、`mkvmerge` がシステムに必要です。パッケージマネージャーでインストールしてください。MKVToolNix: [ダウンロード](https://mkvtoolnix.download/downloads.html)。

**初回実行は遅くなります** — TensorRT エンジンがお使いの GPU 向けにコンパイルされます。これには **15〜60 分** かかります。\
コンパイル中は他のアプリケーション（ブラウザを含む）をすべて閉じ、PC を使用しないでください。\
エンジンは `model_weights` フォルダにキャッシュされ、以降の実行で再利用されます（エンジンファイルとフォルダを新しいバージョンにコピーできます）。

**最新の Nvidia ドライバーをインストールしてください。**\
テスト済みドライバー: **591.67**（ただし 59x 系ならどれでも OK で、最低必要バージョンです）。\
**Jasna には最低コンピュート能力 7.5 の GPU が必要です**

### 検出モデル
一般的に最新の rf-detr を選択することを推奨します。\
Lada Yolo モデルは 2D アニメーションの処理に優れているため利用可能です。

### セカンダリ復元モデル
Jasna/Lada はモザイク領域の 256×256 ピクセルクロップを取得し、256×256 解像度のまま復元します。そのため、モザイク領域が大きい場合（クローズアップ、4K 動画など）はぼやけた結果になります。\
これを緩和するために、256×256 を 512×512 または 1024×1024 にアップスケールする 2 番目の復元モデルを使用できます。
現在サポート:
- **RTX Super-resolution**（非常に高速、まあまあの品質）。非常に高速、無料、依存関係ゼロ。一部の動画ではフリッカーが発生する場合があります — まず短いクリップでテストしてください。Jasna を英語のみの文字のフォルダに配置してください。
- **TVAI**（最高品質、最も遅い）。[Topaz Video](https://www.topazlabs.com/topaz-video)（有料、Windows のみ）が必要です。推奨モデル: **iris-2**。\
  ```--tvai-args``` でモデルやその他のパラメータをカスタマイズできます。デフォルトは iris-2 です。\
  "Topaz Video" の環境変数を設定してください:\

### 最大クリップサイズ + テンポラルオーバーラップ
テンポラルオーバーラップの主な目的は、クリップ境界でのフリッカーを軽減することです。\
20 を超えてもあまり効果はありません。オーバーラップが大きいほど処理時間は長くなりますが、フリッカーは少なくなります。\
**可能な限り大きいクリップサイズを選び、オーバーラップを 8〜20 に設定してください。**\
限られたテストからのガイダンス:
- クリップサイズ 60 + テンポラルオーバーラップ 6
- クリップサイズ 90 + テンポラルオーバーラップ 8
- クリップサイズ 180 + テンポラルオーバーラップ 15（BasicVSR++ コンパイル有効時は 12 GB 以上の VRAM が必要、無効時はより少ない）。

4K 動画はより多くの VRAM を使用します — より小さいクリップサイズでも同等の品質が得られ、はるかに高速に処理できる場合があります。少し実験してみてください。

クリップサイズ 60 未満でも見た目は問題ない場合がありますが、動画によります。モデルコンパイルを無効にしてでも 60 を使うことを推奨します。
```--enable-crossfade``` はフリッカーを軽減するため、常に有効にすべきです。

### 復元モデルのコンパイル
復元モデルは TensorRT サブエンジンにコンパイルされます。\
初回コンパイルには **15〜60 分** かかります — 他のアプリケーション（ブラウザを含む）をすべて閉じ、PC を使用しないでください。\
エンジンはキャッシュされ自動的に再利用されます。パフォーマンスを犠牲にしてコンパイルをオプトアウトできます。

以下の表は **コンパイル済みエンジンのみ** が占める VRAM 量です（総処理 VRAM ではありません）:

| | クリップ 60 | クリップ 180 |
|---|---|---|
| **エンジン VRAM（コンパイル済み）** | 約 1.9 GB | 約 5.4 GB |
| **エンジン VRAM（コンパイルなし）** | 約 1.2 GB | 約 1.2 GB |

### ストリーミング

ファイル全体を処理することなく、復元された動画をリアルタイムで視聴できます。

**ブラウザプレーヤー（内蔵）**\
`--stream` フラグを付けて jasna を実行します（現在は CLI のみ）。HLS プレーヤーのブラウザウィンドウが開きます。動画ファイルを選んで視聴を開始します。シークに対応しています。
```
jasna --stream
```

**Stash 連携**\
Jasna を [Stash](https://github.com/stashapp/stash) 内で直接使用できます — 任意のシーンを再生すると、Jasna によりリアルタイムで処理されます。Stash が Jasna を自動的に起動します。Jasna 対応のカスタム Stash フォークが利用可能です:\
👉 **[Stash v0.30.1-jasna](https://github.com/Kruk2/stash/releases/tag/v0.30.1-jasna)**

セットアップ:
1. 上記リンクから Stash フォークをダウンロードします。
2. Stash を起動する前に環境変数を設定します:
   - `JASNA_CLI_PATH` — `jasna-cli.exe` のフルパス
   - `JASNA_WORKING_DIR` — `jasna-cli.exe` があるフォルダのフルパス
3. Stash を起動して任意のシーンを再生します — Jasna が視聴中にリアルタイムで処理します。シーク対応。

### 免責事項
Jasna は開発初期段階であり、主な目標は復元品質、モザイク検出、速度、VRAM 消費の改善です（この優先順位で）。
そのため、現在このプロジェクトはより技術的なユーザーを対象としており、一部のユーザーにはアクセスしづらい場合があります。
重要な機能により多くの時間を割くためにそうしています。協力していただける方はプルリクエストを歓迎します。

## ビルド
以下のライブラリを ```uv pip install . --no-build-isolation``` でインストールしてください。\
以下の Nvidia ライブラリをビルドするには VS Build Tools 2022 (C++) も必要です。
cmake と ninja があることを確認してください ```uv pip install cmake ninja```\
また、システムに CUDA 13.0 が必要です。

### ソースからの実行（開発者セットアップ）
- `ffmpeg` + `ffprobe` をインストールし、PATH に追加してください（**ffmpeg のメジャーバージョンは 8 である必要があります**）。
- `mkvmerge`（MKVToolNix の一部）をインストールしてください: [ダウンロード](https://mkvtoolnix.download/downloads.html)。
